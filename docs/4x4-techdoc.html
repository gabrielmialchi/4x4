§01

Arquitetura Geral do Sistema

Network

O jogo opera de forma "Serverless" utilizando o Firebase Realtime Database como barramento de estado (State Bus). Para contornar a falta de um backend customizado na fase Alpha, adota-se uma arquitetura onde o Jogador 1 (P1) atua como o Host Autoritativo.

P1 — Host
Monitora inputs de ambos. Processa combate, visibilidade, colisões. Escreve a "Verdade Absoluta" no Firebase.
P2 — Client
Lê a UI baseada no estado Firebase. Empurra apenas seus inputs locais: target, skill, roll, ready.
Firebase
Sincroniza estado via onValue em tempo real. Ambos os clientes observam as mesmas mudanças.
⚠ Limitação Alpha
Esta arquitetura coloca lógica de jogo no cliente P1, abrindo brecha para manipulação. Consulte a Seção 5 (Débito Técnico) para o plano de migração.
§02

Gestão de Estado

State Management

Todo o estado serializável do jogo reside no objeto window.S. O Firebase garante a sincronia em tempo real via onValue.

window.S — JSON Schema (Parcial)
// Estado global sincronizado via Firebase window.S = { players: [ // [0] = P1, [1] = P2 { pos: [r, c], // Array de posição [row, col] skill: "", // "" | "BLOCK" | "TRAP" | "SPRINT" combatSkill: "", // Snapshot no momento do processamento inv: { BLOCK: int, // Quantidade restante TRAP: int, SPRINT: int }, permMod: int, // Acumulador de Engine Building history: [[r,c], ...] // SVG de Aftermath } ], blocks: [{ r, c, owner }], // Blocos no tabuleiro traps: [{ r, c, owner }], // Armadilhas (invisíveis ao oponente) spentRes: [ // Cemitério de recursos { r, c, type, owner, status } // status: wasted | broken | triggered | jumped ], phase: "planning" // State Machine: "planning" | "combat" }
Campo Tipo Descrição
players[n].permMod int Acumulador de modificadores permanentes (Engine Building). Soma de todos os recursos gastos.
players[n].history Array<[r,c]> Histórico de posições para renderização do trajeto no SVG de Aftermath.
spentRes[n].status string enum wasted · broken · triggered · jumped. Usado para UX no Aftermath.
phase string Controle da State Machine principal. Valores: "planning" e "combat".

Lógica de Névoa de Guerra

Fog of War

Calculada puramente no Frontend (Client-side) na função renderBoard(). Distância ortogonal via Manhattan Distance.

🔴 Vulnerabilidade de Segurança
A Fog of War é aplicada apenas na UI — o Firebase entrega o objeto S completo para ambos os jogadores. Posições das armadilhas ocultas do adversário são visíveis via DevTools. Resolução planejada na migração para Backend Real (§05).
§04

Resolução de Combate

Combat Flow

O combate é segmentado em fases assíncronas para permitir feedback visual de rolagem de dados, respeitando o timing percebido pelo jogador.

PASSO 01
Trigger de Colisão

hostProcessTurn() detecta colisão de destinos (same && !arrEq(p1.target, p1.pos)). Muda phase para "combat".

PASSO 02
Rolagem de Dados

Jogadores enviam seu D6 via rollDie(). Ambos aguardam o outro completar antes do processamento.

PASSO 03
Delay de UX

Host detecta ambos roll > 0 e engatilha setTimeout de 2.8s para visualização da matemática na UI — tensão dramática intencional.

PASSO 04
Resolução Final

hostResolveCombat() computa roll + permMod + turnMod. Em empate: ready resetado, itens não consumidos. Em vitória: move vencedor, consome itens, aumenta permMod de ambos, registra falhas no spentRes.

hostResolveCombat() — Score Calculation
const score = (p) => p.roll + p.permMod + p.turnMod; if (score(p1) === score(p2)) { // Empate — nenhum item consumido resetReady(); } else { const winner = score(p1) > score(p2) ? p1 : p2; movePlayer(winner); consumeItems([p1, p2]); // Ambos perdem o item usado incrementPermMod([p1, p2]); // Engine Building acumula logSpentRes(loser); // Cemitério para Aftermath }
§05

Escalabilidade & Débito Técnico

Roadmap Alpha → Beta

Para escalar de Alpha para Beta/Produção comercial, as seguintes refatorações arquiteturais são recomendadas para o Tech Director responsável pela transição.

01
Migração para Servidor Autoritário Real

Atualmente, P1 processa a lógica de jogo — abre brecha para trapaça via manipulação de cliente. Substituir por servidor Node.js utilizando Colyseus ou Nakama como runtime de jogo autoritativo.

Segurança Alta Prioridade Node.js
02
Separação State / View

Substituir a manipulação de DOM procedural (innerHTML, appendChild) por um framework reativo como React, Vue ou Svelte. Alternativa: portar lógica de jogo para Unity/Godot, aproveitando o GDD e as regras amplamente testadas no código Vanilla.

Arquitetura Manutenibilidade React / Godot
03
Cullagem de Dados (Data Culling)

Atualmente, o Firebase entrega o objeto S completo para ambos os jogadores — incluindo posições das armadilhas ocultas do adversário (visíveis via DevTools). O Backend futuro deverá retornar Payloads parciais específicos para P1 e P2, com a Fog of War aplicada server-side.

Segurança Fog of War Server-side
